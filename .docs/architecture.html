pi-fzf-files: dependency-free fzf-style file autocomplete for Pi

This extension replaces Pi's default @ file-reference autocomplete with a TypeScript implementation that builds a local in-memory path index, searches it with fzf-style query syntax, and uses persistent frecency data as a stable ranking tie-breaker.

Entry point: index.ts Runtime: Pi extension API Storage: in-memory index + JSON frecency No runtime dependencies beyond Node/Pi
Runtime lifecycle Local index Search and ranking Frecency Autocomplete integration

System overview

The extension is intentionally small: it keeps all searchable file metadata in memory for low-latency keystroke searches, persists only user reference history, and delegates final text insertion to Pi's built-in autocomplete provider.

Session start Load project frecency JSON, construct FileIndex, wrap the active autocomplete provider, then begin indexing.
→
User types @... The provider extracts the active fzf query and asks FileIndex.search() for the top matches.
→
User submits prompt The input hook scans submitted text for real @path references and records matching paths in frecency.

Fast hot path

Search runs over a cached array of indexed paths and keeps only the best K results in a heap, avoiding a full sort of large repositories.

Portable storage

The file index is not serialized. Only frecency is persisted as project-scoped JSON under Pi's agent directory, written atomically.

fzf-compatible syntax

Queries support fuzzy subsequences, exact matches, boundary matches, prefix/suffix/equal terms, inverse terms, AND groups, and OR groups.

Main modules

Each source file owns one piece of the extension pipeline. The modules communicate through plain TypeScript interfaces and simple data structures.

ModuleResponsibilityImportant implementation details
index.ts Package export. Re-exports the default extension factory from src/extension.ts for Pi package discovery.
src/extension.ts Pi integration and session runtime. Registers lifecycle hooks, wraps autocomplete, tracks prompt input, exposes /fzf-files commands, and coordinates indexing/frecency shutdown.
src/file-index.ts In-memory project path index and search. Walks the repository with opendir(), filters heavy directories, stores IndexedPath[] plus a path set, and ranks matches with TopK.
src/fzf.ts fzf query parser and matcher. Parses extended tokens into OR groups of AND terms; scores fuzzy/exact/boundary/prefix/suffix/equal matches.
src/frecency.ts Persistent reference history. Stores count and timestamps in project-scoped JSON; debounces writes; scores by log frequency times exponential recency decay.
src/provider.ts Autocomplete provider wrapper. Intercepts active @ tokens, returns fzf file suggestions, and delegates non-@ behavior and completion application to Pi's current provider.
src/prefix.ts @ token parsing. Finds the current autocomplete prefix and extracts submitted references, with support for quoted paths and escaping.
src/path-utils.ts Path normalization and quoting. Converts to display paths, normalizes submitted references to project-relative paths, and emits @path or @"path with spaces".
src/top-k.ts Bounded result selection. Maintains a min-heap of the current worst-to-best candidates and sorts only the retained K results for display.

Runtime lifecycle

The extension runtime is scoped to the active Pi session and current working directory. Session replacement and shutdown abort indexing and flush frecency before releasing state.

Runtime shape
{ cwd, frecency, index, rebuildPromise }
Status key
fzf-files status updates show indexing progress, entry counts, truncation, and failures.
Startup
session_start clears any old runtime, flushes old frecency, loads the new store, creates the new index, registers the provider, and starts a rebuild.
Shutdown
session_shutdown clears runtime first, aborts the index walk, and awaits frecency flush.
Commands
/fzf-files stats, reindex, and clear-frecency/reset-frecency.
type Runtime = {
  cwd: string;
  frecency: FrecencyStore;
  index: FileIndex;
  rebuildPromise: Promise<void> | undefined;
};
Stale rebuild protection: rebuild completion checks that the same runtime and promise are still current before updating UI. This prevents aborted indexing from an old session from overwriting the new session's status.

Local file index

The local index is deliberately in-memory. It is rebuilt on session startup or manual reindex, making searches fast without maintaining a persistent database or watcher.

Indexed data model

Every file or directory is represented as an IndexedPath. Directories are stored without a trailing slash in path, but with a slash in matchPath so queries can match directory continuations.

interface IndexedPath {
  path: string;           // project-relative display path, e.g. "src/fzf.ts"
  matchPath: string;      // same path, with trailing slash for directories
  lowerMatchPath: string; // lower-cased cached form reserved for matching use
  name: string;           // basename used as suggestion label
  isDirectory: boolean;
  depth: number;          // slash count for tie-breaking
}

Index storage

  • entries: IndexedPath[] is the searchable ordered snapshot.
  • paths: Set<string> supports O(1) validation when submitted references are recorded.
  • Stats are derived from the entries array: files, directories, truncation, indexing state, timestamp, and duration.
  • No index file is written to disk. A new Pi session rebuilds from the current filesystem state.

Walking behavior

  • Uses Node's async opendir() iterator rather than spawning fd.
  • Default cap: 120,000 entries.
  • Default max depth: 30.
  • Ignores non-file, non-directory entries and skips heavyweight directories such as .git, node_modules, dist, build, .next, coverage, target, and virtualenv/cache directories.
Trade-off: because the index is not watched, filesystem changes made after startup are not visible until /fzf-files reindex or a new session. This keeps implementation and runtime cost low.

Search, matching, and ranking

Search composes fzf-style textual match quality with frecency as a tie-breaker. The textual score dominates so history only changes ordering between otherwise comparable matches.

Parse query parseFzfQuery() tokenizes spaces, escapes, AND terms, OR groups, and inverse markers.
→
Scan snapshot FileIndex.search() walks the current entries array and checks cancellation every 1024 candidates.
→
Keep top K TopK retains only the best suggestions, then returns them best-first for the dropdown.

sbtrktFuzzy subsequenceFinds a subsequence, tightens the span with a backward pass, then scores length, consecutive characters, boundaries, slash boundaries, basename, prefix, and span compactness.
'wildExact substringUses case-insensitive indexOf(), with bonuses for boundary start and basename placement.
'wild'Boundary exactRequires both start and end boundaries. Boundary checks use original target casing so camelCase transitions such as fooBar are meaningful.
^musicPrefixMatches the whole relative path prefix.
.mp3$SuffixMatches the whole relative path suffix.
^file$EqualRequires exact path equality.
!fireInverse exactRejects a candidate when the inner term matches. fzf-style inverse defaults to exact matching.
foo barANDAll non-inverse terms in a group must match; their scores are added.
foo | barORThe best matching group wins.

Final ranking comparator

Within the top-K heap, candidates are ordered by textual score first, then frecency, then deterministic path traits.

1. Higher fzf match score
2. Higher frecency score
3. Directories before files
4. Shallower paths before deeper paths
5. Shorter paths before longer paths
6. Lexicographic path order

Autocomplete provider integration

The extension wraps Pi's active autocomplete provider instead of replacing the whole editor integration. This keeps slash commands and completion application behavior consistent with Pi.

When it intercepts

extractAtFzfPrefix() scans the current line before the cursor for the last valid @ token start. It ignores email-like uses such as a@b.com and avoids @ characters inside double-quoted text.

When it delegates

If no active @ prefix exists, the wrapper calls current.getSuggestions(). If an @ prefix exists but produces no matches, it intentionally returns null so Pi's default file finder does not appear.

Suggestion shaping

  • Search limit is 20 suggestions.
  • Completion value is emitted as @path or quoted @"path with spaces".
  • Directory labels receive a trailing slash and directory completion values keep the slash so the user can keep completing inside the directory.
  • Descriptions include the relative path and, when positive, freq N.NN for the frecency score.

Frecency tracking and persistence

Frecency records which indexed files the user actually referenced in submitted prompts. It is persisted per project and applied during future searches.

Default root
~/.pi/agent/fzf-files/frecency-v1/
Override
PI_CODING_AGENT_DIR changes the Pi agent root.
File name
<sha256(cwd).slice(0, 24)>.json
Schema
{ version: 1, cwd, entries: Record<path, { count, firstAccessed, lastAccessed }> }
Max entries
20,000, pruned by most recent access and then count.
Write strategy
Debounced by 750ms, chained through savePromise, written to a temp file, then atomically renamed.

Recording flow

  1. The input hook receives submitted prompt text before agent processing.
  2. extractAtReferences() finds unquoted @path and quoted @"path with spaces" references.
  3. candidateReferencePaths() normalizes candidates relative to the current project root and strips common trailing punctuation.
  4. The first candidate present in FileIndex.paths is recorded.
  5. FrecencyStore.record() increments count, updates lastAccessed, marks the store dirty, and schedules a save.

Score formula

Frecency combines frequency and recency with a 14-day half-life. A file referenced many times remains useful, but stale history decays naturally.

const age = Math.max(0, now - entry.lastAccessed);
const recency = Math.pow(0.5, age / HALF_LIFE_MS);
const score = Math.log2(entry.count + 1) * recency;
Why JSON instead of SQLite? The searchable file index is already in memory, and frecency lookups are O(1) map reads. JSON keeps installation dependency-free while still supporting atomic saves and pruning.

Correctness, cancellation, and edge cases

The implementation favors predictable behavior under cancellation and session changes while keeping per-keystroke work bounded.

Cancellation

  • Index rebuilds own an AbortController.
  • abort() cancels walks and clears indexing state.
  • Search returns empty results when the autocomplete signal is aborted.
  • Stale rebuild promises are ignored before touching UI state.

Path safety

  • Submitted references normalize to project-relative paths.
  • References outside cwd are rejected.
  • Quoted references support escaped characters.
  • Only paths present in the local index update frecency.

Determinism

  • Ranking uses deterministic tie-breakers after score and frecency.
  • Stats are derived from the active entry snapshot.
  • Malformed frecency files are treated as empty stores.

Known non-goals

  • No persistent path index.
  • No filesystem watcher.
  • No cross-project aggregation.
  • No external fzf, fd, or sqlite runtime dependency.
